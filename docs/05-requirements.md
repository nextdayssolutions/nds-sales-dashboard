# 営業ダッシュボード 要件書

## プロジェクト概要

社内向け営業活動支援ダッシュボード。個人の営業生産性を高め、管理者が全体を俯瞰できるツール。

## 想定利用者

- **規模**: 最大20名（初期）→ 将来的に50〜100名
- **ロール**:
  - `admin`（管理者）: 全機能 + 従業員管理
  - `manager`（マネージャー）: 配下メンバーのデータ閲覧
  - `member`（従業員）: 自分のデータのみ

## 技術スタック

| 項目 | 採用技術 |
|---|---|
| フロントエンド | Next.js 14 (App Router) + TypeScript |
| UIライブラリ | Tailwind CSS + shadcn/ui |
| 認証 | Supabase Auth（Google Workspace SSO） |
| データベース | Supabase (PostgreSQL) |
| ホスティング | Vercel (Hobby プラン) |
| 外部API | Google Calendar API, Google Sheets API |

## デザイン要件

**重要**: `docs/01-ui-member-dashboard.jsx` および `docs/02-ui-admin-dashboard.jsx` のデザインに準拠してください。

### デザインシステム

- **背景色**: `#080c18`（ダークネイビー）
- **アクセントカラー**:
  - プライマリ: `#00D4FF`（シアンブルー）
  - 成功: `#00E5A0`（ミントグリーン）
  - 警告: `#FFB830`（アンバー）
  - エラー: `#FF6B6B`（コーラルレッド）
  - セカンダリ: `#B794F4`（パープル）
- **フォント**: DM Sans + Noto Sans JP
- **ボーダー半径**: 10〜20px（角丸を多用）
- **グリッド背景**: 薄い格子状の背景パターン
- **グロー効果**: 画面四隅に radial-gradient のグロー

### UIパターン

- カード型レイアウト（`rgba(255,255,255,0.04)` 背景 + 薄いボーダー）
- KPIカードは4列グリッド（モバイルは2列）
- タブ切り替えは pill 型ボタン
- モーダルは backdrop blur 付き
- アニメーションは `cubic-bezier(0.4,0,0.2,1)` 中心

## 機能要件

### フェーズ1（最優先）

#### F1-1. 認証機能
- Google Workspace SSO のみ（メール/パスワード認証は無効）
- ログアウト機能
- セッション管理（7日間）

#### F1-2. 管理者による従業員招待
- 管理者画面から「従業員を招待」ボタン
- メールアドレス + 氏名 + 部署 + ロール を入力
- Supabase Auth の招待メール自動送信
- 招待されたユーザーは Google アカウントでログイン時に自動アクティベート

#### F1-3. CRM 顧客管理
- **一覧画面**:
  - ステータスでフィルタ（全て / 既存 / 商談中 / 見込み）
  - 関係値スコアの可視化（10段階のドット）
  - 担当者名・業界・最終接触日の表示
- **詳細画面**:
  - 顧客情報の表示・編集
  - 導入商材のタグ表示
- **登録・編集**:
  - 新規顧客登録フォーム
  - 既存顧客の編集
- **権限**:
  - member: 自分が owner の顧客のみ閲覧・編集可
  - manager: 配下メンバーの顧客も閲覧可（編集不可）
  - admin: 全データの閲覧・編集可

### フェーズ2

#### F2-1. 売上管理
- 月次売上入力（商材別、ストック/ショット区分）
- 年間累積売上の自動計算
- 目標 vs 実績のグラフ表示
- 商材別売上の内訳

#### F2-2. カレンダー連携
- Google Calendar API で予定を取得
- 予定と顧客の紐付け
- 当日・今週の予定表示

### フェーズ3

#### F3-1. 自己管理シート
- 理念ビジョン（オレンジ）
- 目標設定（青）
- 育成プログラム（白）
- 進捗の可視化

#### F3-2. PDCAシート
- 期間ごとの Plan / Do / Check / Action 記録
- マネージャーコメント機能
- 振り返り履歴

#### F3-3. 管理者ダッシュボード
- 全社売上集計
- 従業員別パフォーマンス一覧
- アラート機能（達成率低下など）

## データモデル

詳細は `docs/03-db-schema.sql` を参照。

主要テーブル:
- `users` - 従業員マスタ
- `customers` - CRM顧客
- `products` - 商材マスタ
- `customer_products` - 顧客と商材の紐付け
- `sales_records` - 売上レコード
- `personal_sheets` - 自己管理シート
- `pdca_records` - PDCA記録
- `calendar_events` - カレンダーキャッシュ
- `audit_logs` - 監査ログ

## 非機能要件

### セキュリティ
- Row Level Security (RLS) を全テーブルで有効化
- API キーは環境変数で管理
- 監査ログで管理者操作を記録
- 退職者は即座に status を `retired` に変更し、ログイン不可にする

### パフォーマンス
- 一覧画面の初期表示 < 1秒
- 20人規模での同時接続を想定（Supabase Free で十分）

### 運用
- GitHub Actions による週次DBバックアップ
- Supabase Free プランの自動停止回避（週次ピング）
- エラーログは Vercel Analytics で監視

## 開発ルール

```
sales-dashboard/
├── app/                    # Next.js App Router
│   ├── (auth)/             # 認証関連
│   ├── (dashboard)/        # メイン機能
│   ├── admin/              # 管理者画面
│   └── api/                # API Routes
├── components/             # 再利用可能なコンポーネント
│   ├── ui/                 # shadcn/ui
│   └── dashboard/          # ダッシュボード固有
├── lib/
│   ├── supabase/           # Supabaseクライアント
│   └── utils.ts
├── types/
│   └── supabase.ts         # Supabase 型定義(自動生成)
├── docs/                   # ← 設計資料(このフォルダ)
└── .env.local
```

### コーディング方針

- TypeScript strict モード
- Server Components を優先、必要な時だけ Client Components
- Server Actions でデータ変更
- エラーハンドリング必須（try-catch + toast通知）
- コメントは日本語OK、変数・関数名は英語

### コミットルール

- 機能単位で小さくコミット
- メッセージは日本語OK（例: 「顧客一覧画面の実装」）
- push 前に必ず `npm run build` が通ることを確認
